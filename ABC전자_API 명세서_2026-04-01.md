### API 명세서

#### 1. 주문 관리 모듈

##### 1.1 주문 접수/수정/취소

- **HTTP Method + URL Path**: 
  - POST /api/orders
  - PUT /api/orders/{orderId}
  - DELETE /api/orders/{orderId}

- **설명**: 고객사별로 주문을 접수, 수정, 취소할 수 있는 기능을 제공합니다.

- **인증 필요 여부**: 예 (고객사 로그인 필요)

- **Request Headers**:
  - Authorization: Bearer {token}
  - Content-Type: application/json

- **Path Params**:
  - orderId (수정 및 취소 시 필요)

- **Request Body (JSON 예시)**:
  - POST /api/orders
    ```json
    {
      "customerId": "CUST123",
      "orderDate": "2023-11-01",
      "items": [
        {"itemId": "ITEM001", "quantity": 10},
        {"itemId": "ITEM002", "quantity": 5}
      ],
      "note": "Urgent delivery",
      "urgentFlag": true
    }
    ```
  - PUT /api/orders/{orderId}
    ```json
    {
      "items": [
        {"itemId": "ITEM001", "quantity": 15}
      ],
      "note": "Change in quantity"
    }
    ```

- **Response Body (성공 + 에러 케이스 JSON 예시)**:
  - 성공 (POST)
    ```json
    {
      "orderId": "ORD12345",
      "status": "접수"
    }
    ```
  - 성공 (PUT)
    ```json
    {
      "orderId": "ORD12345",
      "status": "수정 완료"
    }
    ```
  - 성공 (DELETE)
    ```json
    {
      "orderId": "ORD12345",
      "status": "취소"
    }
    ```
  - 에러
    ```json
    {
      "error": "Invalid order data",
      "code": 400
    }
    ```

- **HTTP Status Codes**:
  - 200 OK (수정, 취소 성공)
  - 201 Created (주문 접수 성공)
  - 400 Bad Request (잘못된 요청 데이터)
  - 401 Unauthorized (인증 실패)
  - 404 Not Found (주문 ID가 존재하지 않음)
  - 409 Conflict (주문 상태 변경 불가)
  - 500 Internal Server Error (서버 오류)

- **비즈니스 규칙 / 유효성 검증**:
  - 주문 접수 시 주문 상태는 '접수'로 설정됩니다.
  - 주문 수정 시 기존 주문 정보를 업데이트하며, 취소 시 주문 상태는 '취소'로 변경됩니다.
  - 주문은 접수 후 24시간 내에 확인되어야 합니다.

##### 1.2 주문 상태 추적

- **HTTP Method + URL Path**: GET /api/orders/{orderId}/status

- **설명**: 주문의 현재 상태를 조회합니다.

- **인증 필요 여부**: 예

- **Request Headers**:
  - Authorization: Bearer {token}

- **Path Params**:
  - orderId

- **Response Body (성공 + 에러 케이스 JSON 예시)**:
  - 성공
    ```json
    {
      "orderId": "ORD12345",
      "currentStatus": "배송",
      "statusChangeLog": [
        {"status": "접수", "changedAt": "2023-11-01T10:00:00Z"},
        {"status": "확인", "changedAt": "2023-11-01T12:00:00Z"},
        {"status": "출고", "changedAt": "2023-11-02T09:00:00Z"},
        {"status": "배송", "changedAt": "2023-11-02T15:00:00Z"}
      ]
    }
    ```
  - 에러
    ```json
    {
      "error": "Order not found",
      "code": 404
    }
    ```

- **HTTP Status Codes**:
  - 200 OK (성공)
  - 401 Unauthorized (인증 실패)
  - 404 Not Found (주문 ID가 존재하지 않음)
  - 500 Internal Server Error (서버 오류)

- **비즈니스 규칙 / 유효성 검증**:
  - 주문 상태는 접수, 확인, 출고, 배송, 완료의 순서로 변경됩니다.
  - 각 상태 변경 시 로그가 기록됩니다.

##### 1.3 대량 주문 엑셀 업로드

- **HTTP Method + URL Path**: POST /api/orders/bulk-upload

- **설명**: 엑셀 파일을 업로드하여 대량 주문을 처리합니다.

- **인증 필요 여부**: 예

- **Request Headers**:
  - Authorization: Bearer {token}
  - Content-Type: multipart/form-data

- **Request Body**:
  - 엑셀 파일 (파일 필드명: file)

- **Response Body (성공 + 에러 케이스 JSON 예시)**:
  - 성공
    ```json
    {
      "successCount": 50,
      "errorRows": [
        {"row": 5, "error": "Invalid item ID"},
        {"row": 12, "error": "Quantity exceeds stock"}
      ]
    }
    ```
  - 에러
    ```json
    {
      "error": "File format not supported",
      "code": 400
    }
    ```

- **HTTP Status Codes**:
  - 200 OK (성공)
  - 400 Bad Request (잘못된 파일 형식)
  - 401 Unauthorized (인증 실패)
  - 500 Internal Server Error (서버 오류)

- **비즈니스 규칙 / 유효성 검증**:
  - 엑셀 파일의 각 행은 하나의 주문으로 처리됩니다.
  - 파일 내 오류가 있을 경우 해당 행은 처리되지 않고 오류 메시지가 반환됩니다.

#### 2. 재고 관리 모듈

##### 2.1 실시간 재고 현황 조회

- **HTTP Method + URL Path**: GET /api/inventory

- **설명**: 품목별 현재 재고를 실시간으로 확인합니다.

- **인증 필요 여부**: 예

- **Request Headers**:
  - Authorization: Bearer {token}

- **Response Body (성공 + 에러 케이스 JSON 예시)**:
  - 성공
    ```json
    {
      "inventory": [
        {"itemId": "ITEM001", "currentStock": 100, "warehouseLocation": "A1"},
        {"itemId": "ITEM002", "currentStock": 50, "warehouseLocation": "B2"}
      ]
    }
    ```
  - 에러
    ```json
    {
      "error": "Unauthorized access",
      "code": 401
    }
    ```

- **HTTP Status Codes**:
  - 200 OK (성공)
  - 401 Unauthorized (인증 실패)
  - 500 Internal Server Error (서버 오류)

- **비즈니스 규칙 / 유효성 검증**:
  - 재고는 입고, 출고 시 실시간으로 업데이트됩니다.
  - 재고가 안전재고 이하로 떨어지면 자동 알림이 발송됩니다.

##### 2.2 입고/출고 이력 추적

- **HTTP Method + URL Path**: GET /api/inventory/transactions

- **설명**: 각 품목의 입고 및 출고 이력을 조회합니다.

- **인증 필요 여부**: 예

- **Request Headers**:
  - Authorization: Bearer {token}

- **Query Params**:
  - itemId (필터링 옵션)

- **Response Body (성공 + 에러 케이스 JSON 예시)**:
  - 성공
    ```json
    {
      "transactions": [
        {"itemId": "ITEM001", "transactionType": "입고", "quantity": 50, "timestamp": "2023-11-01T10:00:00Z", "handledBy": "USER123"},
        {"itemId": "ITEM001", "transactionType": "출고", "quantity": 20, "timestamp": "2023-11-02T14:00:00Z", "handledBy": "USER124"}
      ]
    }
    ```
  - 에러
    ```json
    {
      "error": "Item not found",
      "code": 404
    }
    ```

- **HTTP Status Codes**:
  - 200 OK (성공)
  - 401 Unauthorized (인증 실패)
  - 404 Not Found (품목 ID가 존재하지 않음)
  - 500 Internal Server Error (서버 오류)

- **비즈니스 규칙 / 유효성 검증**:
  - 각 입출고 시점에 대한 기록을 저장하며, 이력은 품목별로 필터링 가능합니다.

#### 3. 정산/회계 모듈

##### 3.1 월별 매출/매입 자동 집계

- **HTTP Method + URL Path**: GET /api/finance/sales

- **설명**: 월별 매출 및 매입을 자동으로 집계하여 제공합니다.

- **인증 필요 여부**: 예

- **Request Headers**:
  - Authorization: Bearer {token}

- **Query Params**:
  - month (예: 2023-11)

- **Response Body (성공 + 에러 케이스 JSON 예시)**:
  - 성공
    ```json
    {
      "month": "2023-11",
      "sales": 1000000,
      "purchases": 500000,
      "customerId": "CUST123"
    }
    ```
  - 에러
    ```json
    {
      "error": "Data not available",
      "code": 404
    }
    ```

- **HTTP Status Codes**:
  - 200 OK (성공)
  - 401 Unauthorized (인증 실패)
  - 404 Not Found (데이터가 존재하지 않음)
  - 500 Internal Server Error (서버 오류)

- **비즈니스 규칙 / 유효성 검증**:
  - 매출 및 매입 데이터는 주문 및 재고 데이터와 연동되어 자동 집계됩니다.

#### 4. 사용자/권한 관리 모듈

##### 4.1 사용자 역할 관리

- **HTTP Method + URL Path**: 
  - POST /api/users
  - PUT /api/users/{userId}/roles

- **설명**: 사용자에게 역할을 부여하고, 역할에 따라 데이터 접근을 제한합니다.

- **인증 필요 여부**: 예 (관리자 권한 필요)

- **Request Headers**:
  - Authorization: Bearer {token}
  - Content-Type: application/json

- **Path Params**:
  - userId (역할 수정 시 필요)

- **Request Body (JSON 예시)**:
  - POST /api/users
    ```json
    {
      "username": "newuser",
      "password": "securepassword",
      "roleId": "ROLE_USER"
    }
    ```
  - PUT /api/users/{userId}/roles
    ```json
    {
      "roleId": "ROLE_ADMIN"
    }
    ```

- **Response Body (성공 + 에러 케이스 JSON 예시)**:
  - 성공 (POST)
    ```json
    {
      "userId": "USER123",
      "status": "Created"
    }
    ```
  - 성공 (PUT)
    ```json
    {
      "userId": "USER123",
      "status": "Role updated"
    }
    ```
  - 에러
    ```json
    {
      "error": "Unauthorized action",
      "code": 403
    }
    ```

- **HTTP Status Codes**:
  - 201 Created (사용자 생성 성공)
  - 200 OK (역할 수정 성공)
  - 400 Bad Request (잘못된 요청 데이터)
  - 401 Unauthorized (인증 실패)
  - 403 Forbidden (권한 부족)
  - 404 Not Found (사용자 ID가 존재하지 않음)
  - 500 Internal Server Error (서버 오류)

- **비즈니스 규칙 / 유효성 검증**:
  - 각 역할은 특정 데이터에 대한 접근 권한을 가집니다.
  - 사용자 역할에 따라 데이터 접근이 제한되어야 합니다.

### 공통 비즈니스 규칙 및 유효성 검증

- 모든 API 요청은 인증 토큰을 필요로 하며, 유효한 토큰이 없을 경우 401 Unauthorized 응답을 반환합니다.
- 주문은 접수 후 24시간 내에 확인되어야 하며, 재고가 안전재고 이하로 떨어지면 자동 알림이 발송됩니다.
- 시스템은 24시간 가동되며, 99.9%의 가용성을 유지해야 합니다.
- 모든 데이터는 일일 백업되어야 하며, 데이터 손실 방지를 위해 철저한 검증이 필요합니다.
